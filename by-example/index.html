MNX-Common by example

Welcome to our overview of the new music notation format currently codenamed MNX-Common (see intro here). Our goal is to show you how MNX-Common documents look and work, by example rather than formal specification. This website is primarily intended for people making music-notation software.

On this page, we present many musical situations — starting from a simple “Hello world” example and getting progressively more complex — to show you exactly how each musical situation is encoded. For each example, you’ll see an image of the desired notation, followed by the encoding in MusicXML and MNX-Common.

For some examples, you can toggle between “Relevant section” and “Full document.” The examples default to “Relevant section,” for clarity, but you can always click “Full document” to view the entire encoding with full context.

IMPORTANT NOTE: The MNX-Common spec is not yet ready for implementation. MNX-Common is still a work in progress, and we hope you’ll get involved with its development. Does anything look weird in these examples? Hard to parse? Confusing? Potentially ambiguous? We want to hear about it. Make sure you’ve joined the W3C Music Notation Community Group, and post your ideas to our GitHub issue tracker.

Some notes about these MNX-Common examples:

  • Note stem directions are generally not encoded in these examples, for brevity. This means the consuming program decides which stem direction to use.
  • Barlines are generally not encoded here. In MNX-Common, a standard barline is assumed to exist at the end of each measure, and a final barline is assumed to exist at the end of the last measure in the document. Thus, you only need to encode barlines if they differ from these assumed defaults.

Time signatures

In MusicXML, each part’s <measure> can contain an <attributes> element with an optional <time>. MNX-Common uses a score-wide <global> element. MNX-Common also uses a “microsyntax” — e.g., 4/4 for common time — for brevity. In both cases, you only specify the time signature when it differs from the previous bar.

Key signatures

In MusicXML, each part’s <measure> can contain an <attributes> element with an optional <key>. MNX-Common uses a score-wide <global> element. In both cases, you only specify the key signature when it differs from the previous bar.

Accidentals

In MusicXML, use <accidental> within a <note> to indicate a rendered accidental. In MNX-Common, use an accidental attribute on <note>.

In both MusicXML and MNX-Common, accidentals are only encoded if they are to be rendered. Notes whose accidental is implied from the key signature (such as the Bb note in bar 2 of this example), or notes repeating a currently active accidental (such as the second Db in bar 2), are not encoded as having an accidental.

Cautionary accidentals, such as the D natural in bar 3 of this example — that is, accidentals that are not technically required to be displayed and serve as a courtesy to the musician — are not treated any differently than “non-cautionary” accidentals.

Ties

MusicXML uses two elements: <tie> to describe the sound and <tied> to indicate a visual tie in notation. The first note gets type="start", the second gets type="stop".

MNX-Common uses a single element: <tied>. Only the first note in the tie encodes this; its target attribute points at the id of the end note.

For a series of notes tied together (e.g., the second tie in the above example), both MusicXML and MNX-Common require you to encode each “inner tie” separately.

Beams

In MusicXML, you put one or more <beam> elements within a <note> element — a separate element for each beam. The number attribute specifies whether it’s the outer (eighth-note) beam, versus an inner (e.g., 16th note) beam. Parsers need to keep track of currently open beams and look for the values continue or end.

In MNX-Common, you wrap all a beam’s notes/rests within a single <beamed> element.

Beams (with hooks)

In MusicXML, you specify beam hooks using the value "forward hook" or "backward hook" within the <beam> element.

In MNX-Common, there’s no way to specify the direction of beam hooks. Beam hooks are expected to be calculated automatically by software.

Tuplets

MusicXML uses two separate concepts: a <time-modification> element to encode the tuplet’s ratio and a <tuplet> element to encode the engraved tuplet itself. These go inside <note> elements. Parsers need to keep track of currently open tuplets and look for the value stop.

MNX-Common uses a single <tuplet> element, which contains the notes or rests therein. The inner attribute encodes the rhythmic duration of the tuplet’s elements as if the tuplet were ignored; the outer attribute encodes the rhythmic duration taking the tuplet’s ratio into account.

Multiple voices

In MusicXML, parsers need to keep track of an internal position; this example uses <backup> to move the internal position back to 0 to begin the second voice. In MNX-Common, voices are explicitly grouped within <sequence> elements.

Octave shifts (8va)

In MusicXML, you put <octave-shift> elements around the notes that are affected by the octave shift; the issue of whether the octave shift affects the first note following the closing <octave-shift> is interpreted differently by different programs.

In MNX-Common, you put a single <octave-shift> element just before the first note it affects. The element’s end attribute specifies the measure location of the last note for which the octave shift applies. In this example, the octave shift’s final note, the C quarter note, has measure location 2:1/2 (measure index 2, metrical position one half note into the bar).

In both MusicXML and MNX-Common, pitches for notes within an octave shift are specified taking the octave shift into account (see MNX-Common’s definition of “written pitch”). In this example, the final two pitches are C7 and C6, despite the notes being rendered on the same staff line.

Slurs

In MusicXML, you put <slur> elements within the first and last notes in the slur. The first has type=start and the last has type=stop. Use the placement attribute to specify whether the slur is above or below the notes.

In MNX-Common, you put a single <slur> element inside the slur’s first <event>. The element’s target attribute specifies the ID of the last event in the slur. (Every <event> element can have an optional ID.) Use the side attribute to specify whether the slur is above or below.

Slurs (for chords)

In MusicXML, if the start and/or end points of a slur are chords, then the slur markup goes inside the first note of the chord.

In MNX-Common, the slur is attached to the <event>, not to the <note>.

Slurs (targeting specific notes)

In MusicXML, slurs are always defined within <note> elements.

In MNX-Common, a slur can optionally specify which specific notes it’s attached to, using the start-note and end-note attributes. The start-note must be within the event that contains the <slur>, and the end-note must be within the target event.

Slurs (incomplete slurs)

In both MusicXML and MNX-Common, slurs aren’t required to end on specific notes. For “incomplete” slurs like this, the two formats have different approaches.

With MusicXML’s graphical approach, use two <slur> elements within the same note — one with type="start" and the other with type="start". Then use the default-x and default-y attributes to position the slur accordingly.

With MNX-Common’s semantic approach, use location="incoming" or location="outgoing" on the note to which the slur is attached.
